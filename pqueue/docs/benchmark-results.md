## Interpretation guide

### Outbox offline submit

**What this measures.** The transport always returns `RetryLater` (network error), so every `submitPost` call fails the live-send attempt and falls through to durable enqueue. This exercises the write-heavy path: `writeAt` (appending the record to the active segment) and `writeFile` (segment rotation + manifest publish when the segment fills).

**`p50_us` / `p90_us` / `p99_us` / `max_us`** are host wall-clock timings on the POSIX in-memory FS. They vary by machine and are useful only for catching structural regressions (O(n) → O(n²)) on a fixed CI host. Do not cite them as device predictions.

**`sim_p99_ms` / `sim_max_ms`** are deterministic and machine-independent — these are the numbers worth publishing. Multiply by 100 for predicted on-device ms at default calibration.

For the offline submit path, `sim_p99_ms == sim_max_ms` is expected. Per-op simulated latency is either ~0.138ms (normal writeAt-only enqueue) or ~3.934ms (rotation: writeAt + 2× writeFile for new segment header and manifest publish). Since ~7% of enqueue operations trigger a rotation at 256B payload (segment holds ~14 records at 280 bytes each including framing), both p99 and max land on the rotation cost. On device: ~393ms worst-case per `submitPost` call.

**`write_amp`** = total bytes written to FS / (N × payload bytes). Values above 1.0 reflect fixed framing overhead: segment header (20B), enqueue event header (16B), CRC+footer (8B), plus manifest publish (~62B amortized per rotation). Write amp decreases as payload grows because framing is a smaller fraction of the write. 2.15× at 256B and 1.30× at 1KB are expected.

**`read_bpp = 0.0`** is a correctness invariant for this scenario — no reads occur on the enqueue-only path. Any non-zero value indicates unexpected read activity and should be treated as a regression.

**`writeAt = N`** is also a hard invariant — exactly one segment append per submitted record. Verified by `--strict`.

**`remove = 0`** because compaction is never triggered (no dead data exists until records are popped). Verified by `--strict`.

**`writeFile`** counts segment creations and manifest writes; it scales with the number of rotations. At 256B payload the store creates ~69 new segments per 1000 records (168 writeFile calls = ~69 segment headers + ~99 manifest writes, with some manifest cache hits reducing writes). At 1KB payload rotations are more frequent because fewer records fit per segment, driving up the writeFile count.
