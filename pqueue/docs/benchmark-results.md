## Interpretation guide

### Enqueue

**What this measures.** A fresh store is pre-mounted, then N records are enqueued one at a time. Each call exercises only the write path: one `writeAt` to append the event to the active segment, plus a `writeFile` pair (new segment header + manifest publish) on every rotation.

**`sim_p99_ms == sim_max_ms` is expected.** Per-op sim latency is either ~0.138ms (writeAt-only) or ~3.934ms (rotation: writeAt + 2× writeFile). At 64B payload, ~22 records fit per segment so ~4.5% of ops rotate; at 2048B, only 1 record fits per segment so every op rotates — explaining the jump in `writeFile` from 44 to 2000 and in `p50_us` from ~17µs to ~85µs. Both p99 and max land on the rotation cost across all payload sizes.

**`write_amp`** decreases with payload size as fixed framing (20B segment header + 24B event overhead) becomes a smaller fraction of the total write. 3.19× at 64B, ~1.10× at 2048B.

**`readAt = 0` and `remove = 0`** are hard invariants verified by `--strict`: the enqueue hot path should never read and should never trigger compaction.

### Outbox offline submit

**What this measures.** The transport always returns `RetryLater`, so every `submitPost` fails the live-send attempt and falls through to durable enqueue. Measures the write-heavy offline path including outbox framing overhead on top of the raw queue enqueue.

**`write_amp`** is higher than raw enqueue at the same payload because the outbox wraps the payload in an HTTP request envelope before enqueuing. At 256B payload, outbox write_amp (2.148×) vs enqueue write_amp (1.559×) reflects that envelope overhead.

**`writeAt = N`, `read_bpp = 0`, `remove = 0`** are hard invariants verified by `--strict`.
