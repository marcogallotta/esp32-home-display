## Interpretation guide

### Enqueue

A fresh store is pre-mounted, then N records are enqueued one at a time. Each call exercises only the write path: one `writeAt` to append the event to the active segment, plus a `writeFile` pair (new segment header + manifest publish) on every rotation.

**`sim_p99_ms == sim_max_ms` is expected.** Per-op sim latency is either ~0.138ms (writeAt-only) or ~3.934ms (rotation: writeAt + 2Ă— writeFile). At 64B payload ~46 records fit per segment so ~2.2% of ops rotate; at 2048B only 1 record fits per segment so every op rotates â€” explaining the jump in `writeFile` from 44 to 2000 and `p50_us` from ~16Âµs to ~77Âµs. Both p99 and max land on the rotation cost across all payload sizes.

**`write_amp`** reflects the ratio of bytes written (segment data + manifest) to payload bytes. It decreases with payload size as fixed framing (20B segment header + 24B event overhead) becomes a smaller fraction of the total write. At 2048B payload every record fills a segment so `write_amp â‰ˆ (20 + 24 + 2048) / 2048 = 1.02`, close to the observed 1.040.

**`readAt = 0`, `read_bpp = 0`, `remove = 0`** are hard invariants verified by `--strict`.

### Peek+pop

N records are pre-loaded into a fresh store (not timed), counters are snapshotted, then N peek+pop pairs are timed. Each `peek` does one `readAt` to fetch the payload; each `pop` does one `writeAt` to append the POP tombstone.

**`sim_p99_ms` and `sim_max_ms` diverge**, unlike enqueue. Most ops cost ~0.483ms (one readAt at 345Âµs + one writeAt at 138Âµs = 483Âµs). Occasionally a pop fills the active segment and triggers a rotation, costing ~4.279ms. The low `writeFile=10` count (vs `writeAt=1000`) shows rotations are rare on the drain path â€” only ~5 occur across 1000 pops at all payload sizes, because the POP tombstone (20B) is far smaller than any payload and many tombstones share a segment.

**`write_amp`** reflects only the POP tombstone (20B) and infrequent rotation overhead, divided by payload bytes. At 64B: â‰ˆ (1000Ă—20 + ~600) / (1000Ă—64) â‰ˆ 0.32. Values decrease linearly with payload size since the 20B tombstone is fixed.

**`read_bpp == payloadBytes`** exactly â€” one full payload read per peek, no more. Verified by `--strict` along with `writeAt == N`, `readAt == N`, `remove == 0`.

### Outbox offline submit

The transport always returns `RetryLater`, so every `submitPost` fails the live-send attempt and falls through to durable enqueue. Measures the write-heavy offline path including outbox HTTP envelope framing overhead on top of the raw queue write.

**`write_amp`** is higher than raw enqueue at the same payload because the outbox wraps the payload in an HTTP request envelope before enqueuing. At 256B payload: outbox 1.249Ă— vs enqueue 1.110Ă—. At 1024B: outbox 1.077Ă— vs enqueue 1.042Ă—.

**`writeAt = N`, `read_bpp = 0`, `remove = 0`** are hard invariants verified by `--strict`.
