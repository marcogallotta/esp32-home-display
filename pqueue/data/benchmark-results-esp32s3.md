# Benchmark Results (POSIX structural)

The numeric baseline is [`benchmark-results-posix.json`](benchmark-results-posix.json), generated by:

```bash
make update-benchmark-baseline
```

CI diffs every run against this baseline using `tools/benchmark_regression.py`. Only
deterministic structural fields are compared (I/O counts, write amplification, read bytes
per record, idle step/noop/hot/cap counts); wall-clock times are excluded.

---

## What this benchmark measures

The POSIX benchmark runs on an in-memory filesystem. It is a **structural regression tool**,
not a device latency predictor. Use it to catch:

- Extra `writeAt` / `readAt` / `writeFile` / `remove` calls introduced by a code change
- Write amplification regressions (`write_amp`)
- Read-per-record regressions (`read_bpp`)
- Idle compaction invariant violations (`hot_compactions > 0`, `cap_exhausted > 0`)

Wall-clock times (`p50_us` … `max_us`) are host-local and machine-dependent. They are
informational only and excluded from CI regression comparisons.

For real device latency numbers, see the on-device tests:
- `env:esp32s3-idle-sanity` — worst-case `compactIdle(1)` step timing on real LittleFS

---

## Key structural findings

All numbers below are from `data/benchmark-results-posix.json`.

**Enqueue** `readAt = 0` on the hot path — confirmed across all payload sizes. `writeAt = N`
exactly. Write amplification ranges from 1.110× (256B) to 1.396× (64B, small payload means
more segment header overhead per byte).

**Peek+pop** `readAt = N`, `writeAt = N`, `read_bpp == payloadBytes` exactly — one readAt
(peek) and one writeAt (POP tombstone) per record, no extra I/O.

**Outbox offline submit** Write amplification slightly higher than raw enqueue due to HTTP
envelope framing (256B: ~1.249× vs 1.110×). `readAt = 0`, `remove = 0` on the enqueue-only path.

**Mount** `writeAt = 0`, `writeFile = 0`, `remove = 0` — clean mount is strictly read-only.
`readAt` scales linearly with pre-loaded record count (one `readAt` per record to replay).

**Idle compaction** `hot_compactions = 0` across all payload sizes — the clean-storage
invariant holds. `cap_exhausted = 0` — the workload stays within store capacity.
