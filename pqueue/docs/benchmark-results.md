## Interpretation guide

### Enqueue

A fresh store is pre-mounted, then N records are enqueued one at a time. Each call exercises only the write path: one `writeAt` to append the event to the active segment, plus a `writeFile` pair (new segment header + manifest publish) on every rotation.

**`sim_p99_ms == sim_max_ms` is expected.** Per-op sim latency is either ~0.138ms (writeAt-only) or ~3.934ms (rotation: writeAt + 2× writeFile). At 64B payload ~46 records fit per segment so ~2.2% of ops rotate; at 2048B only 1 record fits per segment so every op rotates — explaining the jump in `writeFile` from 44 to 2000 and `p50_us` from ~16µs to ~77µs. Both p99 and max land on the rotation cost across all payload sizes.

**`write_amp`** reflects the ratio of bytes written (segment data + manifest) to payload bytes. It decreases with payload size as fixed framing (20B segment header + 24B event overhead) becomes a smaller fraction of the total write. At 2048B payload every record fills a segment so `write_amp ≈ (20 + 24 + 2048) / 2048 = 1.02`, close to the observed 1.040.

**`readAt = 0`, `read_bpp = 0`, `remove = 0`** are hard invariants verified by `--strict`.

### Peek+pop

N records are pre-loaded into a fresh store (not timed), counters are snapshotted, then N peek+pop pairs are timed. Each `peek` does one `readAt` to fetch the payload; each `pop` does one `writeAt` to append the POP tombstone.

**`sim_p99_ms` and `sim_max_ms` diverge**, unlike enqueue. Most ops cost ~0.483ms (one readAt at 345µs + one writeAt at 138µs = 483µs). Occasionally a pop fills the active segment and triggers a rotation, costing ~4.279ms. The low `writeFile=10` count (vs `writeAt=1000`) shows rotations are rare on the drain path — only ~5 occur across 1000 pops at all payload sizes, because the POP tombstone (20B) is far smaller than any payload and many tombstones share a segment.

**`write_amp`** reflects only the POP tombstone (20B) and infrequent rotation overhead, divided by payload bytes. At 64B: ≈ (1000×20 + ~600) / (1000×64) ≈ 0.32. Values decrease linearly with payload size since the 20B tombstone is fixed.

**`read_bpp == payloadBytes`** exactly — one full payload read per peek, no more. Verified by `--strict` along with `writeAt == N`, `readAt == N`, `remove == 0`.

### Outbox offline submit

The transport always returns `RetryLater`, so every `submitPost` fails the live-send attempt and falls through to durable enqueue. Measures the write-heavy offline path including outbox HTTP envelope framing overhead on top of the raw queue write.

**`write_amp`** is higher than raw enqueue at the same payload because the outbox wraps the payload in an HTTP request envelope before enqueuing. At 256B payload: outbox 1.249× vs enqueue 1.110×. At 1024B: outbox 1.077× vs enqueue 1.042×.

**`writeAt = N`, `read_bpp = 0`, `remove = 0`** are hard invariants verified by `--strict`.

### Mount

A setup store enqueues N records into a temp directory, then a fresh store instance mounts the same directory. Only the `mount()` call is timed; setup is excluded. Fixed 256B payload; the store is sized for 1000 records so no setup enqueue fails regardless of which N is under test.

**`sim_p99_ms == sim_max_ms`** because there is exactly one mount per run; all 5 repeats collapse to the same simulated cost (simulated latency is deterministic).

**`sim_p99_ms` scales linearly with pre-loaded record count**: empty=2.070ms, 100 records=75.550ms, 1000 records=729.638ms. Each record requires one `readAt` to replay the segment (282 bytes read per record = 256B payload + 24B event overhead + 2B rounding). On device: empty mount ≈ 207ms, 100-record ≈ 7.6s, 1000-record ≈ 73s.

**`write_amp = 0`, `writeAt = 0`, `remove = 0`** are hard invariants verified by `--strict`. Clean mount is read-only; no manifest publish or tail truncation fires on a store that was closed cleanly.

### Idle compaction

Models the intended application pattern: enqueue a burst, drain 90%, then compact idle to completion, repeat for 3 cycles. Only `compactIdle(1)` steps are timed; enqueue and drain cost is excluded.

**`hot_compactions = 0`** is the key correctness invariant, verified by `--strict`. It confirms the clean-storage invariant: `compactIdle` between cycles eliminates all dead data before the next burst, so the write path never stalls for compaction.

**`idle_steps = 10, idle_noops = 3`** across 3 cycles means 7 productive compaction steps and 3 terminating noOps (one per cycle). The store fully compacts in ~2–3 steps per cycle at these workload parameters.

**`max_step_sim_ms`** is the primary on-device latency figure:
- 256B payload: 27.535ms × 100 = **~2.8s** worst single step
- 492B payload: 48.533ms × 100 = **~4.9s** worst single step

The 492B result matches the on-device calibration run (Run 4: simMaxLatency=48.6ms × 100 = 4860ms, actual=4861ms).

**`total_idle_sim_ms`** per repeat (3 cycles combined):
- 256B: 90.994ms × 100 = **~9.1s** total idle compaction budget
- 492B: 154.446ms × 100 = **~15.4s** total idle compaction budget
